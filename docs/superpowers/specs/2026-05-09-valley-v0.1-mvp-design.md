# Valley v0.1 (MVP) — Design Document

**Date:** 2026-05-09
**Status:** Draft, awaiting user review
**Owner:** frederickjerusha
**Phase:** 1 of 3

---

## 1. Goal

Build the foundation of **Valley** — a fast, lightweight, AI-aware desktop terminal for macOS, with a classic-terminal aesthetic (Gruvbox-signature) and an integrated AI side-panel. The MVP must be a daily-usable terminal on its own, before the deeper AI features (Phase 2) and editor / preview / cross-platform work (Phase 3) land.

Valley is conceptually inspired by an existing terminal (terax-ai), but is independently designed and implemented. None of terax-ai's source is modified or redistributed; valley's code is original. Where terax-ai has proven a pattern works (PTY-over-Tauri-channels, OSC-based shell integration, keychain-backed BYOK), valley adopts the *pattern*, not the code.

## 2. Non-goals (Phase 1)

These ship in Phase 2 or 3 and are explicitly out of scope here:

- Code editor (CodeMirror) and AI-inline edits / diff
- Web preview of dev servers
- MCP (Model Context Protocol) support
- Inline ghost suggestions inside the prompt
- "Ask valley?" error banner under failing commands
- Hover command explainer
- Voice input
- Providers beyond OpenAI + Anthropic (Google, Groq, xAI, Cerebras, LM Studio defer to Phase 2)
- Background-process spawn / log streaming
- Workflows / saved-command palette
- Linux + Windows packaging
- Auto-updater
- Sub-agents and complex agentic loops (Phase 2)
- Chat session sharing / sync

If any of these creep into the plan, push back.

## 3. Success criteria

The MVP is done when, on macOS:

1. `pnpm tauri dev` boots in under 1 s and renders the zen-default screen.
2. `pnpm tauri build` produces a `.app` bundle ≤ **12 MB** (terax sits at ~7 MB; we have headroom for AI side-panel components but must stay sub-12).
3. A user can open valley, type `ls`, get output rendered via xterm webgl, with a working zsh prompt.
4. Cmd+T opens a new tab. Cmd+D / Cmd+Shift+D split the active tab vertically/horizontally. Cmd+arrow navigates panes.
5. The chrome auto-hides on idle and reveals on hover/keystroke (zen mode).
6. The user can paste an OpenAI or Anthropic key in Settings → AI; it is written to the macOS keychain. The key never appears in any settings file or `localStorage`.
7. Cmd+I toggles the AI side-panel. Cmd+K opens the omnibar.
8. The user can ask the AI a question; the AI can call `read_file`, `list_directory`, `write_file`, and `run_command` tools. Write/run pause for an in-UI approval card.
9. The signature Gruvbox theme is the default; user can switch to Gruvbox Light Hard, Tokyo Night Storm, or Nord in settings.
10. A `VALLEY.md` at workspace root is loaded and injected into the AI's system context.
11. Sessions persist across restarts via `tauri-plugin-store`.
12. `pnpm exec tsc --noEmit` and `cargo clippy -- -D warnings` both pass.

Identical to terax-ai's stack. Justification: the stack is already best-in-class for small AI-native terminals; deviating would cost engineering time we should spend on differentiation.

## 6. High-level architecture

### Two-process model (mirrors terax)

```
┌─ Rust process (src-tauri) ──────────────┐    ┌─ Webview (React) ───────────┐
│ owns all OS access                       │    │ never touches FS/PTY direct │
│                                          │    │                              │
│ Tauri commands (invoke()):              │◀──▶│ @/lib/native.ts wraps invoke│
```

## 7. Detailed component design

### 7.1 PTY subsystem

**Responsibility:** spawn an interactive shell (zsh by default on macOS) bound to a Tauri-allocated session id, stream output to the webview, accept input + resize signals.

**Tauri commands:**

```rust
// id is a UUID generated frontend-side; rust treats it as opaque string.
async fn pty_open(
    state: State<PtyState>,
    id: String,
    shell: Option<String>,            // override; default = $SHELL or zsh
    cwd: Option<String>,
    cols: u16, rows: u16,
    on_event: Channel<PtyEvent>,      // streams PtyEvent::Output | Exit
) -> Result<(), String>;

async fn pty_write(state: State<PtyState>, id: String, data: String) -> Result<(), String>;
async fn pty_resize(state: State<PtyState>, id: String, cols: u16, rows: u16) -> Result<(), String>;
async fn pty_close(state: State<PtyState>, id: String) -> Result<(), String>;

#[serde(tag = "type")]
enum PtyEvent {
    Output { bytes: String },   // base64 from rust, decoded in pty-bridge
    Exit { code: Option<i32> },
}
```

**State:**

```rust
struct Session {
    pair: PtyPair,             // portable-pty
    writer: Box<dyn Write + Send>,
    child: Box<dyn Child + Send + Sync>,
}

pub struct PtyState(RwLock<HashMap<String, Session>>);
```

**Shell init injection (`shell_init.rs`):** before spawning, write the bundled `zshrc.zsh` / `bashrc.bash` into a per-session temp directory and set `ZDOTDIR=<tmp>` / `BASH_ENV=<tmp>/bashrc.bash` so the user's normal config still sources but valley's OSC-emitting hooks layer on top. Init scripts emit:
- **OSC 7** (`\e]7;file://<host><path>\a`) — cwd reporting after each prompt
- **OSC 133** A/B/C/D — prompt markers (start of prompt, start of input, pre-execution, post-execution exit code)

The frontend's `osc-handlers.ts` parses these to drive the cwd indicator, prompt-aware navigation, and the future "Ask valley?" error-banner trigger (Phase 2).

### 7.2 Terminal frontend

**`TerminalStack.tsx`** keeps one mounted `Terminal` per tab. Hidden tabs use `className="invisible pointer-events-none"` rather than unmounting — keeps PTYs streaming and avoids reflow on tab switch.

**`Terminal.tsx`** mounts xterm with WebGL renderer, the four addons, the active theme, and Geist Mono. Cleans up on unmount; resizes via `addon-fit` on container resize and forwards `pty_resize` to Rust (debounced 50ms).

**Splits (`splits.ts`):** each tab owns a pane tree:

```ts
type PaneTree =
  | { kind: 'leaf', sessionId: string }
  | { kind: 'split', dir: 'h' | 'v', a: PaneTree, b: PaneTree, ratio: number };
```

Cmd+D → split active leaf vertically (right). Cmd+Shift+D → horizontally (below). Cmd+arrow → focus neighbor. Cmd+W → close active leaf (and collapse parent split).

### 7.3 Zen-mode chrome

The default state hides:
- Tab strip (revealed on hovering top 6px or pressing Cmd+1..9)
- Sidebar (revealed via Cmd+B, persists once open)
- AI panel (revealed via Cmd+I, persists once open)
- Status bar (always visible but only ~18px tall, monochrome)

Title bar uses macOS `TitleBarStyle::Overlay` with hidden title; traffic lights sit over the terminal. Vibrancy is opt-in via Settings → General → Window background → Vibrancy/Solid.

### 7.4 AI subsystem (Phase 1 scope)

**Providers:** OpenAI and Anthropic only. Provider + model selection via Settings → AI. Default: Anthropic `claude-sonnet-4-6` (current latest non-Opus default for cost/speed balance). Document in `config.ts` how to add a provider — Phase 2 will follow the same pattern.

**Agent loop (`lib/agent.ts`):**

```ts
import { Experimental_Agent, stepCountIs } from 'ai';
import { tools } from '../tools/tools';

export function buildAgent(opts: { apiKey: string; provider: 'openai'|'anthropic'; model: string; systemPrompt: string }) {
  return new Experimental_Agent({
    model: providerFor(opts.provider, opts.apiKey).languageModel(opts.model),
    system: opts.systemPrompt,
    tools,
    stopWhen: stepCountIs(MAX_AGENT_STEPS),  // 12 in Phase 1
  });
}
```

**Tools (`tools/tools.ts`):**

| Tool | Auto-execute? | Notes |
|---|---|---|
| `read_file` | Yes | path required; refuses on `security.deny(path)` |
| `list_directory` | Yes | path optional, defaults to active workspace cwd |
| `write_file` | **No** — `needsApproval: true` | Approval card shows path + diff preview |
| `run_command` | **No** — `needsApproval: true` | Approval card shows command + cwd |

`security.ts` deny-list (applied on **both** read and write paths):
```
patterns: ['**/.env', '**/.env.*', '**/.ssh/**', '**/credentials*', '**/*.pem', '**/id_rsa*']
```
Bypassing security must require the user to explicitly disable it per-session in Settings (Phase 2 — Phase 1 has no escape hatch, by design).

**Sessions:** persisted via `tauri-plugin-store` at `valley-sessions.json` (list + `activeId` + per-session `messages:<id>` keys). Title auto-derived from first user message. Switching the API key wipes the chat map; sessions persist on disk.

**Live context bridge:** `App.tsx` calls `setLive({ getCwd, getTerminalContext })` so tools can read the active terminal's cwd + last 300 lines. Lazy — never pre-snapshot.

**`VALLEY.md` integration:** at AI panel mount (and on file-change watcher in Phase 2), `lib/agent.ts` reads `<workspace>/VALLEY.md` if present and prepends its content to the system prompt under a clear delimiter. Keep simple in Phase 1 — no caching, just read on each agent build.

**AI omnibar (Cmd+K):** floating single-line input centered top-third of window. Submitting routes to a transient session named `omnibar:<timestamp>` (not visible in the session list). The first response renders as a card next to the omnibar; pressing Cmd+I "promotes" the omnibar conversation into the side-panel as a real session.

### 7.5 Settings window

A separate webview window (`open_settings_window` Tauri command, mirroring terax). Fixed 720×520, non-resizable, macOS overlay title bar. Sections (left rail) and form (right) — Gruvbox-styled custom toggles, not shadcn defaults.

### 7.6 Theming

Themes are color tokens + xterm color schemes + UI semantic mappings, not just terminal colors.

```ts
type ValleyTheme = {
  id: 'gruvbox' | 'gruvbox-light' | 'tokyo-night-storm' | 'nord';
  ui: { bg, surface, surfaceElevated, borderSubtle, borderStrong,
        textPrimary, textMuted, accentPrimary, accentDanger,
        accentSuccess, accentInfo };  // CSS vars
  xterm: { foreground, background, cursor, ansi[16] };
};
```

`ThemeProvider` writes CSS vars to `:root` and updates the active xterm instance.

### 7.7 Settings persistence

Two stores:
- **Keychain** — only API keys. Service `valley`, account = `<provider>:apikey` (e.g. `openai:apikey`).
- **`tauri-plugin-store`** at `valley-settings.json` — non-secret prefs (theme id, font size, default provider id, default model id, vibrancy on/off, AI panel pinned, sidebar pinned, splits visible).

Sessions live in a separate store file `valley-sessions.json`.

## 8. Data flow examples

### A user types in the terminal
```
keystroke → xterm onData → pty-bridge.write → invoke('pty_write', { id, data })
        → Rust pty::pty_write → portable-pty writer
shell output → portable-pty reader thread → Channel.send(PtyEvent::Output{bytes})
        → frontend Channel handler → xterm.write(decoded)
```

### A user asks the AI to fix a failing build
```
Cmd+I → AI panel open → user types → composer submit
  → useChat (AI SDK v6) → DirectChatTransport
  → buildAgent(...).run({ messages, getLiveContext: () => ({ cwd, terminalTail }) })
  → model decides to call run_command "pnpm build"
  → tool needsApproval: true → UI renders ToolCallCard
  → user clicks Approve
  → tools/tools.ts.run_command via invoke('shell_run_command', { cmd, cwd })
  → Rust shell::shell_run_command spawns child, captures stdout/stderr to ring buffer, returns
  → result fed back into the agent loop
  → agent continues until stepCountIs(MAX) or done
```

## 9. Error handling

- Rust commands return `Result<T, String>` with human-readable messages. Frontend `lib/native.ts` wraps each `invoke` in a typed function that throws on error.
- PTY exit propagates as `PtyEvent::Exit{code}`; `pty-bridge` prints `[process exited with code N]` in the dim muted color, marks the tab as dead but leaves it visible (user can `Cmd+R` to respawn — Phase 1: just close + reopen).
- AI tool errors (denied path, command failed) surface as inline tool-result messages with red accent and a "retry" button. Never silently swallow.
- Settings store corruption → fall back to defaults, log warning, don't crash.

## 10. Testing strategy (Phase 1)

Pragmatic, not exhaustive:

- **Rust unit tests** for `pty::session` (spawn → write → read → close cycle), `secrets` (set → get → delete roundtrip with macOS keychain mocked via feature flag), `fs::tree` (read non-existent dir, symlink loops).
- **Frontend type-check** as a hard gate (`pnpm exec tsc --noEmit` in CI).
- **Manual smoke checklist** kept in `docs/qa/smoke.md`: 10 acceptance scenarios covering each success criterion in §3. Run before tagging a build.
- No e2e (Playwright/WebDriverIO) in Phase 1 — Tauri's e2e story is rough and not worth the time at MVP. Revisit Phase 3.
- **Bundle size check** in CI (custom script): `du -sk src-tauri/target/release/bundle/macos/*.app | awk '{ if ($1 > 12288) exit 1 }'`.

## 11. Performance budgets

| Metric | Budget |
|---|---|
| `.app` bundle | ≤ 12 MB |
| Cold start to first paint | ≤ 800 ms |
| Idle RAM (1 tab, no AI active) | ≤ 180 MB |
| Tab-switch latency | ≤ 16 ms |
| Keystroke → render in xterm | ≤ 32 ms |

Measured manually on the dev machine (M-series Mac); not gated in CI yet.

## 12. Distribution

- macOS only at MVP. `.dmg` build via `pnpm tauri build` with the universal target (`aarch64-apple-darwin` + `x86_64-apple-darwin`).
- Code-signed using a personal Apple Developer ID if user has one; if not, ship unsigned and document the right-click-Open workaround. Notarisation is Phase 3.
- No auto-updater in Phase 1. New builds replace the .app manually.
- Source not published yet — repo is private; OSS license decision deferred to Phase 3.

## 13. Risks & open questions

- **Vercel AI SDK v6 churn.** Lock to `ai@^6.0.x`. If a v7 lands during Phase 1, freeze.
- **Anthropic SDK provider compatibility.** Verify `@ai-sdk/anthropic` supports tool approval flow with Claude Sonnet 4.6 — if not, fall back to OpenAI-only at MVP and document.
- **Geist Mono Variable size.** ~80 KB woff2; acceptable. Subset to Latin Extended at build time.
- **Keychain prompts on macOS.** First key write triggers a system prompt; document this in the Settings → AI screen as expected behavior.
- **xterm WebGL renderer flakiness on certain GPUs.** Have a `useFallbackRenderer` setting that toggles to canvas. Phase 2 if reports come in.

## 14. What "done" looks like for Phase 1 (recap)

A signed-off MVP means:
- All 12 success criteria in §3 pass on a fresh macOS user account.
- Bundle ≤ 12 MB, idle RAM ≤ 180 MB, cold start ≤ 800 ms.
- The four themes work, the design system in `docs/design/` matches the running app within reasonable fidelity.
- `cargo clippy -- -D warnings` and `tsc --noEmit` are green.
- The smoke checklist passes end-to-end.
- A user who has only ever used terax-ai cannot pattern-match valley's UI at first glance: identity (logo, palette, typography, layout) reads as its own product.

## 15. Phase 2 + 3 (preview only — separate specs later)

**Phase 2 — AI depth**
MCP server support; agentic loops with sub-agents; inline ghost suggestions (using AI completions in the prompt buffer); "↩ ask valley?" error banner driven by OSC 133 D exit-code parsing; hover command explainer; voice input; remaining providers (Google, Groq, xAI, Cerebras, LM Studio).

**Phase 3 — Polish & cross-platform**
CodeMirror editor module + AI inline edits + diff; web preview; workflows / saved-command palette; `tauri-plugin-updater`, `tauri-plugin-autostart`, `tauri-plugin-window-state`; Linux + Windows packaging; notarisation; OSS release.
